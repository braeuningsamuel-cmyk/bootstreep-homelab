## 2. Performance Findings

### ‚ö†Ô∏è Bottlenecks
1. **Loki + Promtail + Grafana stack** ‚Äî no log retention configured. Default Prometheus retention is 15 days. If you want long-term observability without storage bloat, configure `compactor.retention_*` on Loki and `prometheus.yml` `retention.time`.
2. **Alloy + Loki `user: "10001:10001"`** ‚Äî explicit UID is correct, but ensure the host has that UID/GID or bind-mount UID mapping.
3. **`uptime-kuma` has no resource limit override** ‚Äî uses the default 4-core / no-memory ceiling. Cap it.
4. **No `pids_limit`** in any compose file ‚Äî runaway container can fork-bomb host kernel.
5. **No `cpuset`** for noisy services (jellyfin, minecraft) ‚Äî they can starve monitoring on the same node.

### ‚úÖ Performance-Positive
- Traefik has explicit CPU+memory limits (good for an always-on proxy)
- Portainer capped at 0.5 CPU / 256M (reasonable)
- Authentik gets 16 resources blocks ‚Äî clearly over-resourced compared to its actual usage profile; check actual consumption

## 3. Bug Findings

### B1: `server_commands.py` line 96 ‚Äî hardcoded `~/docker` is wrong for non-root agents
The `docker_action()` function looks for compose files under `~/docker/<service>/compose.yml`. If the bot runs as a non-root user whose home isn't `/root` or `/home/<user>`, this path is wrong. Fix: read from `os.environ.get("DOCKER_ROOT", Path.home() / "docker")` so it can be overridden in the `.env`.

### B2: `server_commands.py` line 99 ‚Äî `compose_dir.iterdir()` not constrained
The function iterates every subdir of `~/docker` looking for `compose.yml`. If an attacker (or accidentally created) folder contains a `compose.yml`, they'll be able to start/stop it. Mitigated by container-name regex check, but should also restrict to a static list.

### B3: `telegram-bot.py` line 219 ‚Äî `docker logs` doesn't restrict by compose root
Unlike `cmd_restart`, `cmd_logs` doesn't check that the container is part of a stack under `~/docker`. An attacker with a valid chat ID could `docker logs --tail 50 some-other-container` running on the host. The `CONTAINER_RE` regex alone is too loose ‚Äî should also verify container is in `~/docker`.

### B4: `telegram-bot.py` line 96 ‚Äî `docker compose -f ... restart` allowed but `up`/`down`/`logs` not via this path
The function allows `docker compose -f <path> restart` but the same path validation is not applied to `up`/`down`. Inconsistent ‚Äî fix or document why restart is the only allowed mutation.

### B5: `requirements.txt` uses unpinned `>=` versions
A future malicious or buggy release of `python-telegram-bot` could break the bot or introduce a vulnerability. Pin exact versions or use `~=` compatible-release operators. **Add `pip-compile` workflow.**

### B6: `Dockerfile` (hermes) ‚Äî running as root in container
The final `FROM node:22-alpine` has no `USER node` directive. The container runs as root, defeating many seccomp profiles. Add `USER node` and `WORKDIR /app` (already present).

### B7: `Dockerfile` (hermes) line 10 ‚Äî `npm ci --only=production` fails if no `package-lock.json`
The `if [ -f package.json ]` check exists but no check for `package-lock.json`. If you ship only `package.json`, `npm ci` will exit non-zero and the fallback silent failure could leave the image half-built.

### B8: `bootstrap.sh` ‚Äî not reviewed in this pass. Action: read & verify.
Standard pattern, but confirm it doesn't `curl | bash` anything external without checksum verification.

### B9: `Makefile` line 38 ‚Äî yamllint points to `$(COMPOSE_DIR)/` but the `Makefile` defines `COMPOSE_DIR := bootstrap/compose`, while 5 compose files in `compose/` (top-level: nextcloud, watchtower, crowdsec, amp, etc.) are NOT linted. Either include both or document the split.
